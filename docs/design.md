# S-MODE 时钟中断字符串定时输出设计文档

## 1. 需求背景

当前 `tg-rcore-tutorial-ch1-clock` 目标是从最小 S 态裸机程序扩展到“基于时钟中断驱动的周期性行为”：

- 在 S-MODE 中建立时钟中断处理机制；
- 每隔 1 秒输出一个字符；
- 最终完整输出 `hello world`；
- 输出完成后，打印“总耗时”。

本设计文档用于指导后续实现，确保功能、结构与验证路径一致。

## 2. 目标与范围

### 2.1 功能目标

- 使用 S-MODE trap 入口接收并处理时钟中断；
- 建立稳定的“下一次时钟中断”重设逻辑，保证周期触发；
- 以 1 秒为周期输出目标字符串中的下一个字符；
- 在最后一个字符输出后，统计并打印总耗时；
- 输出完成后，不再重复输出字符串，但系统时钟中断机制保持可用。

### 2.2 非目标

- 不实现多任务调度器；
- 不引入用户态应用与系统调用框架；
- 不扩展复杂驱动，仅聚焦定时输出与最小可验证内核路径。

## 3. 现状分析

结合当前仓库状态：

- `src/main.rs` 已具备最小裸机入口与 `panic_handler` 框架；
- `rust_main` 当前为空，需要承接初始化逻辑；
- 项目基于 RISC-V64 裸机执行环境，已有 SBI 能力用于基础输出与关机；
- 当前尚未看到已接入的 S-MODE trap/timer 业务逻辑，预计需要在本项目中补齐。

## 4. 总体设计

### 4.1 设计原则

- 中断最小化：中断处理流程保持短小、可预测；
- 状态显式化：所有定时输出进度通过结构化状态维护；
- 时间一致性：基于统一时间基准进行“秒级”判定；
- 易验证：串口输出可直接观察、可比对。

### 4.2 核心流程

系统启动后流程如下：

1. 进入 `rust_main`；
2. 完成时钟与中断初始化；
3. 初始化字符串任务状态（目标串、索引、起始时刻、完成标记）；
4. 进入主循环（`wfi`/空转等待中断）；
5. 每次时钟中断：
   - 先设置下一次中断；
   - 计算是否达到“1 秒边界”；
   - 若达到且未完成，输出下一个字符；
   - 若输出完成，打印总耗时并置完成标记；
6. 后续中断仅维持系统时钟机制，不再重复业务输出。

### 4.3 状态机设计

定义三态：

- `Idle`：初始化前；
- `Running`：按秒输出中；
- `Done`：字符串已全部输出，耗时已打印。

状态转换：

- `Idle -> Running`：初始化完成；
- `Running -> Done`：最后一个字符输出并完成耗时打印；
- `Done` 保持终态（不再输出字符）。

## 5. 数据结构设计

建议维护一个全局任务状态（放置于内核可变静态区或受控同步封装）：

- `message: &'static [u8]`：固定为 `b"hello world"`；
- `index: usize`：下一个待输出字符位置；
- `start_tick: u64`：首次进入 `Running` 的时间戳；
- `last_emit_tick: u64`：上一次输出字符的时间戳；
- `done: bool`：是否已完成；
- `elapsed_tick: u64`：完成时记录总 tick。

补充时间参数：

- `ticks_per_sec: u64`：平台时钟频率；
- `tick_interval: u64`：时钟中断基础步长（可等于更细粒度，如 10ms/100ms）；
- `emit_period_tick: u64`：字符输出周期，固定为 `ticks_per_sec`。

## 6. 时钟中断与时间计算设计

### 6.1 中断节拍策略

为了稳定“每秒输出一次”，建议采用两层节拍：

- 底层中断可较高频（例如 10ms 或 100ms）；
- 业务输出通过比较 `now_tick - last_emit_tick >= emit_period_tick` 触发。

这样可降低“仅 1Hz 中断导致误差累计与调试困难”的风险。

### 6.2 下一次中断设置

在每次时钟中断入口中，优先调用“设置下一次时钟中断”的逻辑，避免因业务处理时间导致丢节拍。

### 6.3 总耗时计算

完成时记录：

- `elapsed_tick = finish_tick - start_tick`；
- 输出展示建议包含：
  - `elapsed_tick`（原始 tick）；
  - `elapsed_sec = elapsed_tick / ticks_per_sec`（秒）。

若后续已有毫秒换算工具，可额外打印毫秒值。

## 7. 模块划分建议

在不破坏当前最小项目结构前提下，建议按职责拆分：

- `main.rs`：启动流程、初始化调用、主循环；
- `trap` 模块：trap 入口与分发；
- `timer` 模块：时钟初始化、下一次中断设置、tick 读取；
- `clock_task` 模块：`hello world` 定时输出状态机与业务逻辑。

如当前仓库规模较小，也可先在 `main.rs` 实现最小闭环，再逐步抽模块。

## 8. 并发与安全约束

- 中断上下文会访问共享状态，需保证可变访问安全；
- 在 `no_std` 环境下优先使用简洁、可审计的临界区方案；
- 严禁在中断中执行长耗时或阻塞行为；
- 仅在必要时输出，避免串口打印拖慢中断路径。

## 9. 验证方案

### 9.1 功能验证

- 启动后观察串口输出：
  - 约每秒输出一个字符；
  - 顺序拼接为 `hello world`；
  - 结束后仅输出一次总耗时信息。

### 9.2 时序验证

- 用宿主机计时粗测总时间；
- 理论值约为 11 秒（`hello world` 共 11 个字符）；
- 允许启动相位与中断量化带来的小幅误差。

### 9.3 回归验证

- 构建成功；
- 保持 panic 路径可用；
- 若项目提供 lint/typecheck 命令，执行并通过。

## 10. 风险与应对

- 风险一：tick 频率获取不准  
  应对：统一从同一来源读取频率参数，不混用常量。

- 风险二：中断重设时机错误导致节拍漂移  
  应对：进入时钟中断后优先设置下一次中断。

- 风险三：共享状态竞争  
  应对：以最小临界区保护状态更新，减少竞态。

- 风险四：输出重复或越界  
  应对：严格按 `index < message.len()` 判定，完成后置 `done`。

## 11. 分步实施计划

1. 梳理并接入 S-MODE trap 与 timer 基础路径；
2. 落地 `clock_task` 状态结构与状态机；
3. 接入每秒输出逻辑与完成耗时统计；
4. 联调串口输出时序；
5. 通过构建与运行验证，修正边界问题。

## 12. 验收标准

- 运行后可按秒输出 `hello world`；
- 输出完成后打印总耗时；
- 不出现重复刷屏、字符乱序、越界输出；
- 代码构建通过，整体行为稳定可复现。
